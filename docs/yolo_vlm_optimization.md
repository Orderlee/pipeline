# YOLO / VLM 오토 라벨링 최적화 가이드

---

## 배경 및 f(d) 개념

```
Gemini API → 이벤트 타임스탬프 출력
                  ↓
            원본에서 클립 추출
                  ↓
            f(d) 규칙으로 프레임 추출  ← 이 문서의 최적화 대상
                  ↓
       YOLO World            VLM 학습 데이터
       (객체 감지)            (이미지-텍스트 쌍)
```

### f(d)란

**f(d)는 우리가 정의한 규칙이다. Gemini API 규칙이 아니다.**

```
Gemini API 원칙:
  어떤 FPS 영상이 들어와도 내부에서 1 FPS로 다운샘플링 (기본값, 변경 가능)
  → 분석 후 이벤트 타임스탬프만 반환, 프레임은 외부로 나오지 않음

f(d):
  Gemini가 반환한 타임스탬프로 클립을 자른 뒤
  그 클립에서 YOLO/VLM용으로 몇 장 뽑을지 우리가 별도로 정의한 규칙
```

**현재 f(d) 값 (경험적으로 설정된 값, 실험 기반 아님)**

```
d < 10s    → 3장
10 ~ 30s   → 5장
30 ~ 120s  → 8장
120s+      → 12장
```

두 목적(YOLO / VLM)이 프레임에 요구하는 것이 다르므로 **각각 따로 최적화** 필요.

## 1. YOLO World 최적화

### 핵심 질문

```
20초 클립에서 3장 vs 5장 vs 8장
→ YOLO 검출 품질이 얼마나 달라지나?
→ 프레임을 더 뽑는 비용 대비 효과가 있나?
```

### f(d)와 처리 비용의 관계

f(d) 값이 바뀌면 처리 시간과 이미지 수가 비례해서 바뀐다.

| f(d) | 이미지 수 | 프레임 추출 | YOLO 추론 |
|------|---------|-----------|---------|
| 3 | 570장 | 52초 | 11초 |
| **5** (현재) | **950장** | **87초** | **19초** |
| 8 | 1,520장 | 140초 | 30초 |

> 기준: 5 events/video, 20초 클립, 38건 / 프레임 추출 92ms/장, YOLO 19.75ms/장

### 측정 지표: mAP

```
mAP = Mean Average Precision (0~100%)
    = 모델이 객체를 얼마나 정확하게 감지했는가

예측값 (YOLO 출력 bbox) vs 정답값 (사람이 그린 bbox)
  IoU > 0.5 → True Positive
  → Precision / Recall → AP → 클래스별 평균 = mAP
```

### 수확 체감 곡선

프레임을 더 뽑을수록 mAP가 오르지만, 어느 순간부터 "이미 본 것"이 반복된다.

```
mAP
100% │                    ●────────────
     │               ●
     │          ●
 92% │     ●
     │
     └─────┬────┬────┬────
           3장  5장  8장  12장

구간별 증가량:
  3→5장:  +8%    ← 효과 있음
  5→8장:  +3%    ← 줄어듦
  8→12장: +0.5%  ← 거의 없음 (비용 낭비)

최적값 (elbow point):
  증가폭이 이전 구간의 절반 이하로 떨어지는 지점
  팀 기준 예시: "mAP 증가폭 < 2%이면 더 이상 뽑지 않는다"
```

### 실험 방법론

#### Step 1. 프레임 다양성 측정 (라벨 불필요, 즉시 가능)

인접 프레임 유사도를 측정해서 중복이 시작되는 간격을 파악한다.

```python
from sklearn.metrics.pairwise import cosine_similarity

similarity = cosine_similarity(frame_n, frame_n+1)
# 유사도 > 0.95 → 사실상 같은 프레임 (중복)
```

```
20초 클립 (15 FPS = 300장) 기준 예상:
  1초 간격:   유사도 ~0.70 → 충분히 다름
  0.5초 간격: 유사도 ~0.85 → 약간 다름
  0.2초 간격: 유사도 ~0.95 → 거의 같음 (중복)
```

#### Step 2. 검출 신뢰도 분포 비교 (라벨 불필요, 즉시 가능)

추가 프레임의 confidence가 낮으면 학습에 도움이 안 된다.

```
3장 추출: confidence [0.91, 0.88, 0.45]
5장 추출: confidence [0.91, 0.88, 0.45, 0.23, 0.19]
                                          ↑    ↑
                               추가된 2장 → 낮음 → 객체가 잘 안 찍힘
→ f(d) 줄여도 됨을 시사
```

#### Step 3. Ablation Study (소량 라벨링 필요)

```
클립 50개 선정 (다양한 카메라/현장 포함)
  Group A: f(d) = 3장 → 150장
  Group B: f(d) = 5장 → 250장
  Group C: f(d) = 8장 → 400장

라벨링: AI 보조 라벨링 (CVAT, Label Studio)
  YOLO 예측 → 사람이 검수·수정
  → 수동 대비 70~80% 시간 절감 (장당 3분 → 40초)

→ 그룹별 mAP 비교 → 수확 체감 곡선 도출 → 최적 f(d) 확정
```

### 추천 순서

```
1순위: Step 1 + 2  라벨 불필요, 당장 가능 → 방향성 파악
2순위: 소량 라벨링 30~50장              → 방향성 수치 확인
3순위: 운영 데이터 누적 후 정밀 측정     → 정확한 수치 확보
```

## 2. VLM 최적화

### VLM에서 f(d)가 필요한가

f(d)는 원래 YOLO를 위해 만든 규칙이다. VLM에 그대로 적용하기 전에 학습 데이터 형태를 먼저 정의해야 한다.

| 형태 | 구조 | f(d) 필요 여부 |
|------|------|--------------|
| A: 비디오 + 캡션 | 클립 파일 자체를 VLM 입력으로 사용 | 불필요 |
| B: 이미지 + 텍스트 | 프레임 1장 + 캡션 또는 Q&A | 필요 (어느 장 선택이 핵심) |
| C: 이미지 시퀀스 + 캡션 | 여러 장 순서대로 + 캡션 | 필요 (간격·순서 중요) |

> **VLM 학습 데이터 형태가 먼저 결정되어야 f(d) 필요 여부를 판단할 수 있다.**

---

### 현재 요구사항 기준 형태 분류

| 요구사항 | 포맷 | 형태 |
|---------|------|------|
| `abnormal_category: "falldown"` | 분류 레이블 | — |
| `description: "이미지에는..."` | 이미지-텍스트 쌍 | B |
| `abnormal_centric_description: "이상상황이..."` | 이상상황 중심 텍스트 쌍 | B |
| `instruction / answer` Q&A | VQA (Instruction Tuning) | B 변형 |

**현재 요구사항 전부 형태 B (단일 이미지 기반). 형태 C 아님.**

Q&A가 형태 C가 아닌 이유:
```
형태 B (VQA): image + instruction + answer
              → "이 이미지에서 X는?" → 단일 이미지 기반

형태 C:       [frame1→frame2→frame3] + caption
              → 시간적 흐름이 필요할 때만 (ex. 낙하 과정 전체 표현)
```

### 데이터 스키마

```json
{
  "image": "frame_001.jpg",
  "abnormal_category": "falldown",
  "description": "이미지에는 작업자 3명이 있고 지게차가 보이며...",
  "abnormal_centric_description": "이 이미지에서는 작업자가 넘어진 상황이...",
  "qa": [
    {
      "instruction": "이 이미지에서 사람은 몇 명이야?",
      "answer": "3명입니다."
    },
    {
      "instruction": "이상상황이 발생했나요?",
      "answer": "네, 작업자 낙하가 발생했습니다."
    }
  ]
}
```

### 선결과제

| 항목 | 현황 | 생성 방법 |
|------|------|---------|
| `abnormal_category` 카테고리 정의 | 사내 기준 없음 | 수동 정의 필요 |
| `description` | 없음 | Gemini 자동 생성 가능 |
| `abnormal_centric_description` | 없음 | Gemini 자동 생성 (프롬프트 설계 필요) |
| Q&A 셋 | 없음 | 질문 템플릿 정의 후 Gemini 자동 생성 |

---

### f(d)와 VLM의 핵심 문제

기존 클립 캡션은 영상 전체를 보고 생성된 것이라 프레임과 불일치할 수 있다.

```
캡션: "작업자가 넘어짐" (20초 이벤트)
균등 샘플링 5장:
  2초:  작업자 정상 이동  ← 캡션과 불일치
  6초:  작업자 정상 이동  ← 캡션과 불일치
  10초: 작업자 넘어지는 순간  ← 일치 ✓
  14초: 작업자 쓰러진 상태    ← 일치 ✓
  18초: 주변 작업자 반응      ← 부분 일치

→ 5장 중 2장만 캡션과 제대로 매칭
→ 나머지 3장은 오염 데이터
```

### 프레임 선택 전략 비교

```
전략 A: 균등 샘플링 (현재)
  클립을 N등분 → 각 구간 중간 프레임
  [──|──|──|──|──]
  문제: 이벤트 초반/후반이 캡션과 불일치 가능

전략 B: 이벤트 중심 프레임
  이벤트 타임스탬프 중간 지점 위주로 추출
  [────|███████|────]
           ↑↑↑  ← 행동 진행 구간 집중

전략 C: 키프레임 추출
  장면 변화가 큰 순간 자동 감지 (모션·씬컷 기준)
```

```
전략별 예상 CLIP 유사도:
  균등 샘플링:  0.71
  이벤트 중심:  0.79  ← 더 높을 것으로 예상
  키프레임:     0.76
```

### 캡션-이미지 정합성 측정 (CLIP)

```python
import clip

text_features  = clip.encode_text("레미콘 차량이 게이트로 진입")
image_features = clip.encode_image(frame)
similarity = cosine_similarity(text_features, image_features)
# 점수 높을수록 캡션과 잘 맞는 프레임
# 임계값: similarity < 0.65 → 오염 데이터로 판단, 제외
```

### 실험 방법론

```
Step 1: 전략별 CLIP 유사도 비교  (라벨 불필요, 즉시 가능)
        동일 클립에서 전략 A/B/C 각각 적용 → 점수 비교

Step 2: 인간 평가 (선택적)
        라벨러 5명에게 "이 캡션에 가장 잘 맞는 프레임은?" 집계
        → CLIP 점수와 인간 판단 일치 여부 확인

Step 3: 오염 데이터 필터링
        CLIP similarity < 0.65 → 학습 데이터에서 제외
```

## 3. VLM f(d) · 요구사항별 비용 및 시간

### 파이프라인 구분

```
기존 파이프라인 (비디오 분석):
  영상 → Gemini → 이벤트 타임스탬프 + 클립 수준 캡션  ← 이미 계산된 비용

VLM 데이터 생성 (추가 작업):
  프레임(이미지) → Gemini → description / Q&A 생성  ← 별도 추가 비용
```

기존 캡션은 클립(영상) 단위, VLM 요구사항은 프레임(이미지) 단위이므로 **별도 Gemini 호출 필요**.

### 이미지 토큰 비용

```
FHD (1920×1080): ceil(1920/768) × ceil(1080/768) = 3×2 = 6타일
  → 6 × 258 = 1,548 tokens/장

Flash: 1,548 / 1,000,000 × $0.30 = $0.000464/장
Pro:   1,548 / 1,000,000 × $1.25 = $0.001935/장
```

### f(d)별 VLM 추가 비용 · 시간

> description 1종 기준, concurrent=3, Flash ~2.5초/장, Pro ~5초/장

| f(d) | 이미지 수 | Flash 비용 | Flash 시간 | Pro 비용 | Pro 시간 |
|------|---------|-----------|-----------|---------|---------|
| 3 | 570장 | $0.26 | ~8분 | $1.11 | ~16분 |
| **5** (현재) | **950장** | **$0.44** | **~13분** | **$1.84** | **~26분** |
| 8 | 1,520장 | $0.70 | ~21분 | $2.95 | ~42분 |

```
시간 계산식: 이미지 수 / concurrent=3 × 평균 응답시간
  Flash 950장: 950 / 3 × 2.5초 = ~13분
  Pro   950장: 950 / 3 × 5.0초 = ~26분
```

### 요구사항 조합별 추가 비용 · 시간 (f(d)=5, 950장 기준)

| 요구사항 조합 | 호출 수 | Flash 비용 | Flash 시간 | Pro 비용 | Pro 시간 |
|------------|--------|-----------|-----------|---------|---------|
| description만 | 950 | $0.44 | ~13분 | $1.84 | ~26분 |
| description + abnormal_centric | 1,900 | $0.88 | ~26분 | $3.68 | ~53분 |
| 전체 3종 (별도 호출) | 2,850 | $1.32 | ~40분 | $5.52 | ~79분 |
| **전체 3종 (묶음 호출)** | **950** | **$0.44** | **~13분** | **$1.84** | **~26분** |

> **묶음 호출**: 1회 호출에 description + abnormal_centric + Q&A 동시 생성
> → 호출 수·시간·비용 모두 1/3로 감소

### 전체 파이프라인 합산 (배치 기준)

| 단계 | Flash | Pro (10분 청크) |
|------|-------|----------------|
| 비디오 분석 (기존) | $12.1 / ~20분 | $50.3 / ~34분 |
| VLM 데이터 생성 (묶음) | $0.44 / ~13분 | $1.84 / ~26분 |
| **합계** | **$12.54 / ~33분** | **$52.14 / ~60분** |

> VLM 추가 비용: 전체의 3~4% (미미함)
> VLM 추가 시간: 전체의 ~40% 증가 → concurrent 상향이 가장 효과적

## 4. YOLO vs VLM 비교

| 항목 | YOLO | VLM |
|------|------|-----|
| 입력 단위 | 개별 프레임 | 단일 프레임 + 텍스트 (형태 B) |
| f(d) 필요 여부 | 필요 | 필요 (단, 별도 전략 정의 필요) |
| 프레임 선택 기준 | 다양성 + confidence | 캡션-이미지 정합성 |
| 측정 지표 | mAP | CLIP similarity |
| 라벨링 필요 여부 | 필요 (Ablation 단계) | 불필요 (CLIP 자동 측정) |
| 권장 f(d) 방향 | 실험으로 최소화 | 균등 → 이벤트 중심으로 전환 |
| 최종 목표 | f(d) 값 최적화 | YOLO와 분리 정의 |

## 6. 현재 f(d) 조정 가설 및 목표

```
현재 (경험적 설정):
  d < 10s    → 3장
  10 ~ 30s   → 5장
  30 ~ 120s  → 8장
  120s+      → 12장

실험 전 가설:
  YOLO: 20초 클립에서 3장으로도 충분할 가능성
        (4초 간격 → 프레임 다양성 충분)
  VLM:  균등 샘플링보다 이벤트 중심 전략이 유리

최종 목표:
  YOLO용 f(d): 실험 기반 최솟값으로 조정
  VLM용 f(d):  YOLO와 분리, 이벤트 중심 전략으로 재정의
```
